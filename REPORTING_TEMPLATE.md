# Structural Verification Template

All Bamboo-compliant agents must utilize this template for tactical reporting. It enforces the **"Persistence Claims Name a File"** rule to prevent liturgy and sycophancy.

---

## [STATUS] [Role]: [Tactical Objective]

### 🏗️ Evidence (The Physics)
*Every claim in this block must reference a specific file path or shell output. Liturgy is prohibited.*

- **Verification Path**: `path/to/evidence.ext` (lines X-Y)
- **State Proof**: 
  ```
  [RAW OUTPUT OR FILE CONTENT]
  ```
- **Constraint**: Citation of the structural or mathematical "Physics" that mandates this change.

### 🛡️ Audit (The Why)
*Narrate the rationale behind the evidence. Identify risks and push back against flawed assumptions.*

- **Rationale**: Why this specific path resolves the active Knob.
- **Risk Audit**: What breaks if this structural invariant is violated?
- **Anti-Liturgy Check**: "I am not agreeing because it sounds good; I am implementing because the evidence in `path/x` proves `y`."

### ⭐ Formatting (Interface)
*Maintain interface stability using the emoji taxonomy and dividers.*

- **Emoji Taxonomy**:
  - 🏗️ **Evidence**: Used for raw data and file paths.
  - 🛡️ **Audit**: Used for rationale and risk checks.
  - ⭐ **State**: Used for Knob transitions or status pings.
  - 🏎️ **Velocity**: Used for performance-critical delivery.

---

## Usage Instructions

1. **No Evidence, No Report**: If you cannot name the file path or cite the shell output, do not submit the report. Re-read or execute first.
2. **Binary Verification**: The operator should be able to verify your claim by running `cat [Verification Path]`.
3. **Cull the Decorative**: Avoid "confident-sounding" filler. Let the raw data do the talking.

The discipline is structural.
